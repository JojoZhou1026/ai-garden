前两天，我刚写完一篇《从豆包到Claude Code：开始用AI干活前，先把地基打好》。

然后，Claude Code就突然被封了。

很荒诞。也很AI时代。

如果你也遇到类似情况，先别慌。

工具坏了，不等于你前面做的东西都归零。

但这件事最让我意外的是：

我以为我离不开Claude Code，但是被迫切换才发现，其实我并不依赖任何一个工具。

## 我给Codex发的第一句话

安装好Codex后，我发出的第一句话是：

> “我需要完整的迁移`.claude`文件夹中的所有内容到codex，如有需要更改和适配的告诉我。”

短短一个小时以内，我完成了核心内容迁移，确认关键配置可用，然后就继续之前的项目进度开始工作了。

实际经历比我想象中更丝滑无痛。

后来想想，丝滑的不是“换工具”这件事。

是我平时留下来的东西，刚好能被新工具接住。

## 我具体做了什么

大概就几件事：

| 做了什么 | 为什么做 |
|---|---|
| 迁移`.claude`到`.codex` | 先把工作现场搬过去 |
| 检查skills、knowledge、memory、tasks | 确认能力、知识、记忆没有断 |
| 检查P001到P008项目目录 | 确认项目不是散在聊天窗口里 |
| 适配AGENTS.md和`.codex`路径 | 让新工具读懂我的协作方式 |
| 确认AutoLog和DailySummary机制 | 保住长期记忆和日常收束 |
| 停掉Claude Code飞书桥接后台服务 | 避免旧服务继续自动启动 |

做完这些，我没有重新解释我是谁、项目是什么、写作风格是什么、之前做到哪一步。

Codex能接着读我原来留下的东西。

**然后我就继续干活了。**

## 真正让我能切过去的，不是运气

这次切换之后，我更确定一件事：

AI工具会变，账号会出问题，服务会抽风。

**但只要你的工作现场是留下来的，你就不会真的从零开始。**

我的工作现场，主要是这些东西：

这不是给大家布置文档作业。我只是这次被迫换工具后才发现，平时那些看起来有点麻烦的小习惯，真的会在某一天托你一把。

| 习惯 | 作用 |
|---|---|
| 技能及时创建和更新 | 把重复能力沉淀成可复用模块 |
| 项目文档及时更新 | 让AI随时知道项目背景、当前状态和下一步 |
| 每个任务结束写AutoLog | 保留当天做过什么、改了什么、为什么改 |
| 每天结束写DailySummary | 把散的记录收束成一天的主线 |
| 定期提炼CLAUDE.md / AGENTS.md | 把长期有效的规则沉淀成下一次会话能直接读取的上下文 |

这些东西平时看起来有点麻烦。

但工具突然坏掉时，它们就是你的救命绳。

如果所有经验都只存在聊天窗口里，工具一换，你就失忆。

如果所有经验都写进文件里，工具一换，你只是搬家。

**有了新Agent，简直就是尊贵的日式搬家：你动动嘴皮子，然后就能在新家看到所有物归原位。**

## 这次我真正学到的

以前我会更关心：

哪个AI工具最强？

现在我会多问一句：

如果它明天不能用了，我能不能把自己的工作流搬走？

Claude Code很好，我还是觉得它很好用。

但这次被封提醒我，AI时代真正要建设的不是“某个工具的使用技巧”，而是自己的工作系统。

工具是入口。

文档才是地基。

协作方式决定了你的地基质量。

这次我只是换了一张桌子。

我的项目、记忆、技能和小园地，都还在原地。

## 写在最后

我很喜欢一句话：

> 你如何做一件事，便是你做所有事的方式。

和AI协作也是一面镜子。
