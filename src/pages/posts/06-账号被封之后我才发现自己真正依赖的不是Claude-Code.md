前两天，我刚写完一篇《从豆包到Claude Code：开始用AI干活前，先把地基打好》。

然后，Claude Code就突然被封了。

很荒诞。也很AI时代。

我没有去猜具体原因。海外AI工具本来就有很多不稳定因素：账号、网络、支付、风控、地区策略，任何一个环节抽风，都可能让你原本顺手的工具突然不能用。

但这件事最让我意外的是：

我没有太慌。

因为我真正依赖的，并不是Claude Code这个工具本身。

## 我给Codex发的第一句话

安装好Codex后，我发出的第一句话是：

![我安装好Codex后发出的第一句话](/images/codex-first-message.png)

这句话很朴素，但它其实说明了我当时的判断：

不用重新开始。

把现场搬过去。

短短一个小时以内，我完成了核心内容迁移，确认关键配置可用，然后就继续之前的项目进度开始工作了。

这才是我觉得“无痛”的地方。

不是没有技术细节，而是技术细节没有把我拦住。

## 我具体做了什么

大概就几件事：

1. 把`.claude`里的关键内容完整迁移到`.codex`
2. 确认skills、knowledge、memory、tasks都还在
3. 检查P001到P008这些项目目录是否完整
4. 把原来写给Claude的部分规则，适配成Codex能读懂的AGENTS.md和`.codex`路径
5. 确认“结束写AutoLog”“下班写DailySummary”这些记忆机制还能继续跑
6. 停掉原来挂在Claude Code上的飞书桥接后台服务，避免它继续自动启动

做完这些，我没有重新解释我是谁、项目是什么、写作风格是什么、之前做到哪一步。

Codex能接着读我原来留下的东西。

然后我就继续干活了。

## 真正让我能切过去的，不是运气

这次切换之后，我更确定一件事：

AI工具会变，账号会出问题，服务会抽风。

但只要你的工作现场是留下来的，你就不会真的从零开始。

我的工作现场，主要是这些东西：

| 习惯 | 作用 |
|---|---|
| 技能及时创建和更新 | 把重复能力沉淀成可复用模块 |
| 项目文档及时更新 | 让AI随时知道项目背景、当前状态和下一步 |
| 每个任务结束写AutoLog | 保留当天做过什么、改了什么、为什么改 |
| 每天结束写DailySummary | 把散的记录收束成一天的主线 |
| 定期提炼CLAUDE.md / AGENTS.md | 把长期有效的规则沉淀成下一次会话能直接读取的上下文 |

这些东西平时看起来有点麻烦。

但工具突然坏掉时，它们就是你的救命绳。

如果所有经验都只存在聊天窗口里，工具一换，你就失忆。

如果所有经验都写进文件里，工具一换，你只是搬家。

## 这次我真正学到的

以前我会更关心：

哪个AI工具最强？

现在我会多问一句：

如果它明天不能用了，我能不能把自己的工作流搬走？

Claude Code很好，我还是觉得它很好用。

但这次被封提醒我，AI时代真正要建设的不是“某个工具的使用技巧”，而是自己的工作系统。

工具是入口。

文档才是地基。

这次我只是换了一张桌子。

我的项目、记忆、技能和小园地，都还在原地。
